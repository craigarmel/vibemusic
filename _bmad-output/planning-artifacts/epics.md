# vibemusic - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for vibemusic (Synthetica), decomposing the requirements from the PRD, UX Design, and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: Lore Generation — System generates artist biography, personality traits, and lyrics from a text prompt via Gemini 3
FR2: Visual Identity — System generates consistent artist avatar images from text prompt via Nano Banana 2 with character consistency across multiple generations
FR3: Real-time Music Session — Artist launches a live session with camera/mic; Lyria RealTime captures ambiance and generates reactive music in real-time with parameter controls
FR4: Video Clip Generation — System combines generated image + audio to produce a 9:16 vertical video clip (10s) via Veo 3.1
FR5: Session Completion & Auto-Publish — End of session triggers final track consolidation by Lyria and auto-publishes to the artist's space
FR6: Fan Feed — TikTok-style vertical scroll interface with auto-playing 9:16 video clips generated by the platform
FR7: Fan Influence — Fans select influence tags ("Plus sombre", "Plus de basses") that are stored and feed back into the next artist session
FR8: Feedback Loop — Fan influence tags are incorporated into the artist's next live session, creating a continuous creation cycle

### NonFunctional Requirements

NFR1: Real-time audio streaming with sub-second parameter response during live sessions
NFR2: Video generation within demo-acceptable timeframes (Veo 3.1)
NFR3: Character consistency across multiple avatar image generations (Nano Banana 2)
NFR4: Mobile-first responsive design for B2C fan feed
NFR5: Dark/neon aesthetic — deep black, cyan/magenta/purple accents, glass morphism, neon glow effects
NFR6: Hackathon-grade simplicity — no auth, no complex DB, in-memory storage, mock data where AI is not required

### Additional Requirements

- Starter template: create-vite (react-ts) + manual FastAPI setup (monorepo)
- Frontend: Vite 8 + React + TypeScript + Tailwind v4
- Backend: Python FastAPI + Uvicorn + httpx async
- State management: Zustand (useStudioStore + useFeedStore)
- Routing: React Router v7 — /studio (B2B), /feed (B2C)
- WebSocket for Lyria real-time audio streaming (bidirectional)
- REST API for all other operations with async handlers
- Long-running tasks (Veo): task ID + polling endpoint
- In-memory storage (storage.py) + local filesystem for media assets
- API response wrapper: { status, data } / { status, message, detail }
- Vite proxy to FastAPI backend during development
- Deployment: Vercel (frontend) + Railway (backend)
- Environment: .env + Pydantic BaseSettings

### UX Design Requirements

UX-DR1: ArtistProfile component — Artist identity card with avatar (neon border glow), name, genre tags, bio, stats (sessions count, tracks count, fan influences count)
UX-DR2: SessionLauncher component — "Start Live Session" CTA button with mic/camera permissions check flow
UX-DR3: LiveSessionView component — Full-screen immersive view with camera feed (centered, neon border), audio visualizer (bottom third, ~30 frequency bars, cyan-to-purple gradient), session timer (monospace, cyan), end session button, mute mic button, fan influences badge
UX-DR4: AudioVisualizer component — Animated real-time frequency/waveform bars with neon glow, reacting to Lyria output
UX-DR5: SessionComplete component — Centered glass morphism card with final track waveform, play button, duration, fan influences applied (tag pills), "Publish to Artist Space" CTA, "Save as Draft" secondary
UX-DR6: TrackGallery component — Grid of track cards (dark glass morphism, waveform visual, track name, duration, date, influence tag count badge)
UX-DR7: FeedView component — TikTok-style full-screen vertical scroll container with auto-playing 9:16 video clips
UX-DR8: FeedCard component — Full-screen video card with bottom overlay (artist avatar, name, track title, duration), right-side floating action buttons (heart, share, influence)
UX-DR9: InfluenceButton component — Larger than other action buttons, pink/magenta gradient fill with glow, lightning bolt icon
UX-DR10: TagSheet component — Bottom sheet (40% height, glass morphism, backdrop blur) with tag grid (3 columns), pre-defined tags as pills, trending tag highlight, "Send Influence" CTA
UX-DR11: ConfirmToast component — "Ton influence a ete prise en compte" toast, auto-dismiss after 3s
UX-DR12: Global design system implementation — Neon Studio theme: colors (#0a0a0a, #00f0ff, #ff00aa, #8b5cf6), Inter font, glass morphism panels, neon borders, gradient overlays, micro-animations

## Epic 1: Project Foundation & Design System

Artists and fans can access a polished, themed application with the Neon Studio aesthetic fully implemented and both B2B and B2C routes functional.

### Story 1.1: Initialize Monorepo with Frontend and Backend Scaffolding

As a **developer**,
I want a fully initialized monorepo with Vite+React+TypeScript frontend and FastAPI backend,
So that the team can start building features immediately.

**Acceptance Criteria:**

**Given** a fresh clone of the repository
**When** I run `npm create vite@latest frontend -- --template react-ts` and set up the backend folder structure
**Then** the frontend starts on port 5173 with `npm run dev` and the backend starts on port 8000 with `uvicorn app.main:app --reload`
**And** Tailwind v4 is configured via `@tailwindcss/vite` plugin with `@import "tailwindcss"` in index.css
**And** Vite proxy forwards `/api` and `/media` to `localhost:8000`
**And** FastAPI has CORS middleware allowing the Vite dev server origin
**And** `.env.example` files exist in both frontend and backend with placeholder variables
**And** `requirements.txt` includes fastapi, uvicorn, httpx, python-multipart, pydantic-settings

### Story 1.2: Implement Neon Studio Design System & App Shell

As an **artist or fan**,
I want a visually stunning dark neon interface,
So that the app feels like a premium AI music studio from the future.

**Acceptance Criteria:**

**Given** the initialized frontend project
**When** I open the application in a browser
**Then** the global theme uses deep black (#0a0a0a) background with cyan (#00f0ff), magenta (#ff00aa), and purple (#8b5cf6) accents
**And** Inter font is loaded for headings (bold, all-caps) and body text
**And** a `Layout.tsx` component wraps all pages with the dark theme and neon styling
**And** glass morphism utility classes are available (backdrop blur, semi-transparent backgrounds, neon borders)
**And** shared components exist: `Slider.tsx`, `LoadingSpinner.tsx`, `Toast.tsx`

### Story 1.3: Configure Routing and Zustand Stores

As a **developer**,
I want React Router and Zustand stores configured,
So that the B2B and B2C views are accessible and state management is ready.

**Acceptance Criteria:**

**Given** the design system is implemented
**When** I navigate to `/studio`

**When** I navigate to `/feed`
**Then** the B2C Fan Feed shell loads

**And** `useFeedStore` is created with initial state (feed items array, influence state)

## Epic 2: Artist Identity Creation

Artists can create a complete AI-generated virtual idol — lore, personality, lyrics, and consistent visual avatar — from a single text prompt.

### Story 2.1: Lore Generation via Gemini 3

As an **artist/label A&R**,
I want to type a creative prompt and receive a generated biography, personality traits, and lyrics,
So that I can build the identity of my virtual idol.

**Acceptance Criteria:**

**Given** the backend is running and Gemini 3 API key is configured
**When** I send `POST /api/artists` with `{ "prompt": "Fille cyborg de Neo-Paris, fait de la techno triste" }`
**Then** the backend calls Gemini 3 via `gemini.py` service and returns `{ "status": "success", "data": { "artist_id": "...", "lore": { "biography": "...", "personality_traits": [...], "lyrics": "..." } } }`
**And** the artist is stored in `storage.py` in-memory dict keyed by `artist_id`
**And** the frontend `LoreGenerator.tsx` component has a text input field and "Generate" button
**And** on submit, the component calls the API, shows a loading state (`is_generating_lore`), and displays the generated lore
**And** the result is saved to `useStudioStore`

### Story 2.2: Avatar Generation via Nano Banana 2

As an **artist/label A&R**,
I want to generate a consistent visual avatar for my virtual idol,
So that the artist has a recognizable face across all content.

**Acceptance Criteria:**

**Given** an artist has been created with lore (Story 2.1)
**When** I click "Generate Avatar" in the `AvatarGenerator.tsx` component
**Then** the backend calls `POST /api/artists/{artist_id}/avatar` which triggers Nano Banana 2 via `nano_banana.py`
**And** the generated image is saved to `media/avatars/{artist_id}_{timestamp}.png`
**And** the API returns `{ "status": "success", "data": { "avatar_url": "/media/avatars/..." } }`
**And** the avatar is displayed in the frontend with a neon border glow
**And** a second generation produces a visually consistent face (same character, different pose/angle)
**And** the loading state `is_generating_image` is shown during generation

### Story 2.3: Artist Profile View

As an **artist/label A&R**,
I want to see my artist's complete profile with lore and avatar,
So that I can review the identity before starting a session.

**Acceptance Criteria:**

**Given** an artist has lore and avatar generated
**When** I view the Studio Dashboard
**Then** the `ArtistProfile` component displays: avatar (200px circle, neon border glow), artist name, genre tags (cyan pills), biography text, personality traits
**And** stats row shows: "Sessions: 0", "Tracks: 0", "Fan Influences: 0" in glass morphism cards
**And** a "Start New Session" CTA button is visible (gradient cyan-to-purple, neon glow)
**And** the layout includes a left sidebar (80px, icon-only nav) with home, sessions, tracks, influences icons

## Epic 3: Live Music Session

Artists can launch an immersive live session where Lyria RealTime generates and evolves music in response to their performance, then produces a final consolidated track.

### Story 3.1: Session Launch with Camera/Mic Activation

As an **artist**,
I want to launch a live session that activates my camera and microphone,
So that Lyria can capture my ambiance for music generation.

**Acceptance Criteria:**

**Given** an artist profile exists on the Studio Dashboard
**When** I click "Start New Session"
**Then** the `SessionLauncher` component requests browser camera and microphone permissions
**And** if permissions are granted, the app transitions to the `LiveSessionView` full-screen immersive interface
**And** if permissions are denied, an error message explains that camera/mic are required
**And** the camera feed is displayed in the center of the screen (rounded-2xl, neon cyan border glow)
**And** a session timer starts counting (monospace font, cyan glow, top center)
**And** the backend receives `POST /api/sessions` creating a new session in storage

### Story 3.2: Lyria RealTime WebSocket Audio Streaming

As an **artist**,
I want Lyria to capture my live ambiance and generate reactive music in real-time,
So that the music evolves as I perform.

**Acceptance Criteria:**

**Given** a live session is active with camera/mic enabled
**When** the session starts
**Then** the frontend opens a WebSocket connection to `/api/audio/stream`
**And** the browser's audio input (microphone) is captured and sent to the backend via WebSocket
**And** the backend `lyria.py` service processes the audio input through Lyria RealTime API
**And** generated audio chunks are streamed back to the frontend via WebSocket
**And** the audio plays in real-time through the browser's audio output
**And** the `useLyriaStream.ts` hook manages the WebSocket lifecycle (connect, send params, receive audio, disconnect)

### Story 3.3: Real-time Audio Visualizer

As an **artist**,
I want to see the generated music visualized in real-time,
So that I can feel the music reacting to my performance.

**Acceptance Criteria:**

**Given** a live session is active with audio streaming
**When** Lyria generates audio
**Then** the `AudioVisualizer` component displays ~30 frequency bars at the bottom third of the screen
**And** bars are colored with a cyan-to-purple gradient with neon glow
**And** bars animate in real-time reacting to the audio output frequency data
**And** the visualizer uses Web Audio API's `AnalyserNode` to extract frequency data from the playing audio stream
**And** animation runs at 60fps without degrading session performance

### Story 3.4: Live Session Controls

As an **artist**,
I want minimal controls during my live session,
So that I can focus on performing without distractions.

**Acceptance Criteria:**

**Given** a live session is active
**When** I look at the session interface
**Then** I see only: session timer (top center), "End Session" button (top right, red/pink), mute mic button (bottom left, glass morphism circle), fan influences badge (bottom right, showing count of received tags with pink glow)
**And** clicking "Mute Mic" toggles microphone input on/off with visual feedback
**And** the fan influences badge updates in real-time as fans send tags
**And** slider controls (Tempo, Intensity) are available but subtle — small panel that expands on hover/tap

### Story 3.5: Session Completion & Track Publication

As an **artist**,
I want the session to produce a final consolidated track that auto-publishes,
So that fans can discover my creation in the feed.

**Acceptance Criteria:**

**Given** a live session is active
**When** I click "End Session"
**Then** the WebSocket connection closes and Lyria generates a final consolidated audio track
**And** the `SessionComplete` component appears: centered glass morphism card with checkmark icon
**And** the final track waveform is displayed (static, cyan/purple gradient)
**And** track details show: track ID, duration, generation timestamp in monospace text
**And** a play button allows previewing the final track
**And** "Fan Influences Applied" section shows tag pills used during the session
**And** "Publish to Artist Space" CTA button triggers `POST /api/clips` saving the track to storage and making it available in the feed
**And** "Save as Draft" secondary link saves without publishing
**And** after publishing, the track appears in the artist's `TrackGallery`

## Epic 4: Video Clip Generation

Artists can generate a 9:16 vertical video clip combining their avatar image and generated music, ready for the fan feed.

### Story 4.1: Video Generation via Veo 3.1

As an **artist**,
I want to generate a short vertical video clip from my avatar and music,
So that I have visual content for the fan feed.

**Acceptance Criteria:**

**Given** an artist has an avatar image and a published audio track
**When** I click "Generate Clip" in the `ClipGenerator.tsx` component
**Then** the frontend sends `POST /api/clips` with `{ "artist_id": "...", "track_id": "..." }`
**And** the backend `veo.py` service calls Veo 3.1 with the avatar image + audio to generate a 9:16 vertical video (10 seconds)
**And** the API returns immediately with `{ "status": "success", "data": { "task_id": "..." } }`
**And** the frontend polls `GET /api/tasks/{task_id}/status` showing a progress indicator
**And** when complete, the video is saved to `media/clips/{clip_id}.mp4`
**And** the video preview plays in the `ClipGenerator` component
**And** the clip is stored in `storage.py` and linked to the artist

### Story 4.2: Track Gallery with Published Content

As an **artist**,
I want to see all my published tracks and video clips,
So that I can manage my artist's content catalog.

**Acceptance Criteria:**

**Given** an artist has published tracks and/or video clips
**When** I view the Artist Profile page
**Then** a `TrackGallery` component shows a grid (2 columns) of track cards
**And** each card displays: dark glass morphism background, small waveform visual, track name, duration, date, and influence tag count badge (pink)
**And** clicking a track card plays the audio or video preview
**And** cards are ordered by most recent first

## Epic 6: Fan Influence & Feedback Loop

Fans can influence the next artist session via interactive tags, creating a continuous co-creation cycle between fans and AI artists.

### Story 6.1: Influence Button & Tag Sheet

As a **fan**,
I want to tap "Influence" and choose a tag to shape the next session,
So that I feel like a co-creator of the artist's music.

**Acceptance Criteria:**

**Given** a clip is playing in the feed
**When** I tap the `InfluenceButton` (larger than other action buttons, pink/magenta gradient with glow, lightning bolt icon)
**Then** the `TagSheet` bottom sheet slides up (~40% screen height, glass morphism, backdrop blur, rounded-t-2xl)
**And** the video behind is dimmed (50% opacity dark overlay)
**And** a tag grid (3 columns) shows pre-defined tags: "Plus sombre", "Plus de basses", "Plus rapide", "Electro vibes", "Ambient", "Voix douce", "Hardcore", "Melodique", "Experimental"
**And** each tag is a glass morphism pill with neon cyan border
**And** selecting a tag fills it with gradient cyan-to-purple
**And** one tag is highlighted as "Trending" with a pink border
**And** tapping "Send Influence" calls `POST /api/feed/influence` with `{ "artist_id": "...", "clip_id": "...", "tag": "Plus sombre" }`

### Story 6.2: Influence Confirmation & Storage

As a **fan**,
I want confirmation that my influence was registered,
So that I know my voice will impact the next session.

**Acceptance Criteria:**

**Given** I selected a tag and tapped "Send Influence"
**When** the API responds successfully
**Then** the tag sheet dismisses
**And** a `ConfirmToast` appears: "Ton influence a ete prise en compte" with a checkmark icon
**And** the toast auto-dismisses after 3 seconds
**And** the backend stores the influence in `storage.py` influence list with artist_id, clip_id, tag, and timestamp
**And** the fan can continue scrolling the feed

### Story 6.3: Feedback Loop — Influences to Next Session

As an **artist**,
I want to see fan influence tags when launching my next session,
So that I can incorporate fan feedback into my performance.

**Acceptance Criteria:**

**Given** fans have submitted influence tags for my artist
**When** I view my Artist Profile
**Then** the fan influences badge shows the total count of pending tags
**And** a "Fan Influences" section lists the top tags aggregated by count (e.g., "Plus sombre: 12 votes", "Plus de basses: 8 votes")
**When** I start a new live session
**Then** the top influence tags are displayed briefly as an overlay at session start
**And** the influence data is sent to Lyria as initial parameters for the new session (e.g., mood = "sombre", bass_level = "high")
**And** after a session completes, the applied influence tags are cleared from the pending queue
